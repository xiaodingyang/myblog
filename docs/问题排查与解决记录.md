## 1. 生产环境页面偶尔白屏

**环境：** 生产环境（线上）  
**现象：** 部分页面偶尔显示完全空白，本地开发环境无此问题  
**影响范围：** 所有前台页面

### 排查过程

1. **检查服务端日志**：Nginx error log 和 PM2 日志均无异常，API 响应正常（200）
2. **检查前端代码**：发现应用**完全没有 React Error Boundary**
3. **分析构建产物**：Umi 使用代码分割，生成大量 `*.async.js` chunk 文件
4. **定位根因**：
   - 当网络波动导致 async chunk 加载失败时，JavaScript 报错
   - 没有 Error Boundary 捕获错误，整个 React 组件树崩溃
   - 页面渲染为空白，用户无任何提示

### 根本原因

**缺少全局错误边界 + 异步 chunk 加载失败 = 白屏**

这是 SPA 应用的经典问题。Umi/Webpack 的代码分割会按路由生成独立的 JS 文件，当用户访问某个页面时动态加载对应的 chunk。如果此时网络不稳定或者新部署替换了旧的 chunk 文件（用户浏览器缓存的 `umi.js` 引用了已不存在的旧 chunk 文件名），加载失败会导致 JavaScript 错误，进而白屏。

### 解决方案

**1. 添加全局 Error Boundary 组件**

```tsx
// src/components/ErrorBoundary/index.tsx
class ErrorBoundary extends Component<Props, State> {
  static getDerivedStateFromError(error: Error) {
    return { hasError: true, error };
  }

  render() {
    if (this.state.hasError) {
      return (
        <div>
          <h2>页面加载出错了</h2>
          <p>可能是网络波动导致资源加载失败，请尝试刷新页面</p>
          <button onClick={() => window.location.reload()}>刷新页面</button>
        </div>
      );
    }
    return this.props.children;
  }
}
```

在 `app.tsx` 的 `rootContainer` 中包裹整个应用：

```tsx
export function rootContainer(container: React.ReactNode) {
  return (
    <ErrorBoundary>
      <ConfigProvider>{container}</ConfigProvider>
    </ErrorBoundary>
  );
}
```

**2. 添加 chunk 加载失败自动重试**

在 `.umirc.ts` 的 `headScripts` 中添加全局监听器：

```javascript
window.addEventListener('error', function(e) {
  var target = e.target || e.srcElement;
  if (target && target.tagName === 'SCRIPT' && /\.async\.js/.test(target.src || '')) {
    var key = 'chunk_retry_' + target.src;
    if (!sessionStorage.getItem(key)) {
      sessionStorage.setItem(key, '1');
      window.location.reload();
    }
  }
}, true);
```

使用 `sessionStorage` 记录已重试的 chunk，防止无限刷新。

### 知识点

- React Error Boundary 只能捕获**渲染阶段**的错误，不能捕获事件处理、异步代码中的错误
- `getDerivedStateFromError` 用于更新 state 触发降级 UI，`componentDidCatch` 用于记录错误日志
- Error Boundary 必须是 class 组件，函数组件暂不支持
- `addEventListener('error', handler, true)` 第三个参数 `true` 表示捕获阶段，能捕获 script 标签加载失败

## 2. 头像上传后被默认头像覆盖

**环境：** 开发环境 + 生产环境  
**现象：** 在后台「个人设置」页面选择图片裁剪后，头像短暂显示然后立刻被默认头像（UserOutlined 图标）覆盖

### 排查过程（使用 Debug 模式）

1. **添加调试日志**：在 `useEffect`、`onChange`、`customRequest`、`render` 等关键点插入 fetch 日志
2. **第一轮日志分析**：
   - `avatarUrl` 状态正确设置为 blob URL 和服务器 URL
   - `useEffect` 没有在上传后重新触发覆盖 `avatarUrl`
   - **排除了 React 状态管理问题**
3. **第二轮日志分析**（添加图片加载测试）：
   - `Avatar` 组件的 `onError` 回调触发 — 图片加载失败
   - `fetch(serverUrl)` 返回 **HTTP 404**
   - **确认是图片 URL 不可访问导致 Avatar 回退到默认 icon**

### 根本原因

上传 API 返回的相对路径（如 `/uploads/20260315/xxx.jpeg`）在开发环境下通过 Umi 代理访问时返回 404。Ant Design 的 `Avatar` 组件在 `src` 图片加载失败时会自动回退到 `icon` 属性（默认头像），导致视觉上"被覆盖"。

时间线：
1. 裁剪确认 → blob URL 预览显示（正常）
2. 45ms 后上传完成 → `avatarUrl` 切换为服务器 URL
3. 浏览器加载服务器 URL → 404 失败
4. Avatar 回退到默认 icon

### 解决方案

**预加载验证**：上传成功后，先用 `new Image()` 预加载服务器图片，只有 `onload` 成功才切换 URL；如果失败则保持 blob 预览。

```typescript
const handleAvatarUpload = async (file: File) => {
  const localUrl = URL.createObjectURL(file);
  setAvatarUrl(localUrl); // 立即显示本地预览

  const res = await request('/api/upload', { ... });

  if (res.code === 0 && res.data?.url) {
    const preloadImg = new Image();
    preloadImg.onload = () => {
      setAvatarUrl(res.data.url); // 服务器图片可用，切换
      URL.revokeObjectURL(localUrl);
    };
    preloadImg.onerror = () => {
      // 服务器图片不可访问，保持 blob 预览
    };
    preloadImg.src = res.data.url;
  }
};
```

同时使用 `beforeUpload` 返回 `false` 阻止 Upload 组件自动上传，完全手动控制上传流程，避免 Upload 内部状态与自定义状态冲突。

### 知识点

- Ant Design `Avatar` 的 `src` 加载失败时会静默回退到 `icon`，不会抛出错误
- `URL.createObjectURL()` 创建的 blob URL 可以立即用于图片预览，无需等待上传
- 使用 `new Image()` 预加载可以在切换 `src` 前验证图片是否可访问
- Upload 组件的 `beforeUpload` 返回 `false` 会阻止自动上传，适合需要完全自定义上传流程的场景

## 3. CI/CD 部署后线上页面空白

**环境：** 生产环境  
**现象：** GitHub Actions 自动部署成功后，线上所有页面显示空白

### 排查过程

1. 对比本地构建和 CI 构建产物的 MD5 — **不一致**
2. 检查 Nginx 缓存配置 — `expires 30d` + `Cache-Control: public, immutable`
3. 分析 Umi 构建文件名 — `umi.js`、`umi.css` 没有 hash，每次内容变化但文件名不变

### 根本原因

Nginx 对 JS/CSS 设置了 `immutable` 缓存策略（30天），浏览器认为文件永远不变，不会向服务器确认。当 CI 部署更新了 `umi.js` 内容但文件名不变时，浏览器使用缓存中的旧版本，导致新旧代码不兼容 → 白屏。

### 解决方案

修改 Nginx 缓存策略：

```nginx
location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf|eot)$ {
    expires 7d;                                    # 30d → 7d
    add_header Cache-Control "public, must-revalidate";  # immutable → must-revalidate
    try_files $uri =404;
}
```

`must-revalidate` 允许浏览器缓存文件，但每次使用前需向服务器确认是否过期（通过 `ETag` 或 `Last-Modified`），确保文件更新后能及时生效。

### 知识点

- `immutable` 表示文件永远不变，浏览器在缓存有效期内绝不会发起请求
- `must-revalidate` 表示缓存过期后必须向服务器验证，配合 `ETag` 使用效率很高
- 最佳实践：文件名带 hash 的资源（如 `app.a1b2c3.js`）可以用 `immutable`；不带 hash 的（如 `umi.js`）必须用 `must-revalidate` 或 `no-cache`

## 4. 百度站点验证失败

**环境：** 生产环境  
**现象：** 百度搜索资源平台提交站点验证时提示「无法连接到您网站的服务器」

### 排查过程

1. 用 `curl` 外部测试 — HTTP 200 正常
2. 检查 Nginx 错误日志 — 出现 "rewrite or internal redirection cycle" 错误
3. 检查服务器防火墙 — **关键发现**

### 根本原因

服务器的 `iptables` + `ipset` 防火墙规则中，`YJ-GLOBAL-INBLOCK` 黑名单包含了多个百度爬虫 IP。百度爬虫无法访问站点，导致验证失败。

### 解决方案

```bash
# 查看被封锁的 IP
ipset list YJ-GLOBAL-INBLOCK

# 移除百度爬虫 IP
ipset del YJ-GLOBAL-INBLOCK <百度爬虫IP>
```

### 知识点

- 百度爬虫 IP 段包括 `220.181.x.x`、`180.76.x.x`、`116.179.x.x` 等
- `ipset` 是高效的 IP 集合管理工具，常与 `iptables` 配合使用
- 部署安全防护时需注意白名单搜索引擎爬虫 IP

## 5. 微信扫码分享无法显示页面

**环境：** 开发环境  
**现象：** 使用 ShareButton 分享功能，微信扫描 QR 码后显示"无法显示该页面"

### 根本原因

QR 码中编码的 URL 是 `window.location.href`，在开发环境下为 `http://localhost:8003/...`，手机无法访问本地开发服务器。

### 解决方案

ShareButton 组件中的 URL 始终使用生产域名：

```typescript
const PROD_ORIGIN = 'https://www.xiaodingyang.art';

function getShareUrl() {
  if (typeof window === 'undefined') return PROD_ORIGIN;
  return window.location.href.replace(window.location.origin, PROD_ORIGIN);
}
```

## 6. QQ 分享功能异常

**环境：** 全环境  
**现象：** 点击 QQ 分享按钮后弹出的页面无法正常显示或分享失败

### 根本原因

使用了已废弃的 `https://connect.qq.com/widget/shareqq` API，该接口不再稳定。

### 解决方案

替换为 QQ 空间分享 API：

```typescript
function shareToQQ(title: string, summary: string, url: string, cover?: string) {
  let qqUrl = `https://sns.qzone.qq.com/cgi-bin/qzshare/cgi_qzshare_onekey?url=${encodeURIComponent(url)}&title=${encodeURIComponent(title)}&summary=${encodeURIComponent(summary)}&site=${encodeURIComponent('若风的博客')}`;
  if (cover) qqUrl += `&pics=${encodeURIComponent(cover)}`;
  window.open(qqUrl, '_blank', 'width=600,height=500');
}
```

## 7. GitHub Actions 部署失败（pnpm 版本不匹配）

**环境：** CI/CD  
**现象：** GitHub Actions 的 "Install frontend dependencies" 步骤失败

### 根本原因

CI workflow 中配置的 pnpm 版本为 8，但本地项目使用 pnpm 10（lockfile v9 格式），两者不兼容。

### 解决方案

更新 `.github/workflows/deploy.yml`：

```yaml
- name: Setup pnpm
  uses: pnpm/action-setup@v4
  with:
    version: 10  # 从 8 改为 10，与本地一致
```

### 知识点

- pnpm lockfile 格式在大版本间不兼容（v6 → lockfile v5，v8 → lockfile v6，v9+ → lockfile v9）
- CI 中应始终指定与本地一致的包管理器版本
- `--frozen-lockfile` 参数会在 lockfile 与 package.json 不一致时直接报错，适合 CI 环境

## 8. 前端开发环境 API 请求 404

**环境：** 本地开发  
**现象：** 前端所有 API 请求返回 404

### 根本原因

`.umirc.ts` 中 proxy 目标从 `http://162.14.83.58` 改为域名后，服务器 Nginx 配置了 HTTP → HTTPS 301 重定向。Umi 开发代理不会自动跟随重定向。

### 解决方案

```typescript
proxy: {
  '/api': {
    target: 'https://www.xiaodingyang.art',  // 直接使用 HTTPS
    changeOrigin: true,
    secure: false,  // 允许代理到自签名/无效证书的 HTTPS
  },
},
```

### 知识点

- 开发代理 `secure: false` 允许连接到不受信任的 HTTPS 目标
- `changeOrigin: true` 会将请求头的 `Host` 改为目标域名，避免虚拟主机匹配失败

## 9. Nginx 上传文件大小限制

**环境：** 生产环境  
**现象：** 上传头像时 Nginx 报错 `client intended to send too large body`

### 根本原因

Nginx 默认的 `client_max_body_size` 为 1MB，上传的图片超过此限制。

### 解决方案

在 Nginx 配置中增大限制：

```nginx
server {
    client_max_body_size 10m;
    # ...
}
```

### 知识点

- `client_max_body_size 0` 表示不限制，但不建议在生产环境使用
- 该配置可以放在 `http`、`server` 或 `location` 块中，作用域不同
- 后端也应做文件大小校验，不能仅依赖 Nginx
