# 首屏性能优化总结

## 一、优化背景

博客首屏加载的 JS 资源总量约 **1.86MB**，其中粒子动画依赖的 Three.js 占了 **973KB（52%）**，严重拖慢首屏渲染速度。用户打开首页时需要等待所有 JS 下载、解析、执行完毕后才能看到内容，体验较差。

### 优化前资源分布

| 文件 | 大小 | 说明 |
|------|------|------|
| `umi.js` | 831KB | 主包（React + Umi + Antd） |
| `7744.async.js` | 973KB | Three.js 粒子动画 |
| `layouts__FrontLayout.async.js` | 23KB | 全局布局 |
| `p__home__index.async.js` | 33KB | 首页组件 |
| **合计** | **~1.86MB** | 首屏需全部加载 |

## 二、优化措施

### 1. 粒子背景延迟加载（减少 973KB 首屏 JS）

**问题：** `ParticlesBackground` 组件依赖 Three.js（973KB），通过静态 `import` 导入到全局布局 `FrontLayout.tsx` 中，导致每个前台页面都会在首屏阻塞加载这个巨大的 JS 文件。

**方案：** 将静态导入改为 `React.lazy` 动态导入，配合 `Suspense` 和 `useEffect` 延迟挂载，让浏览器先完成首屏内容渲染，再在空闲时加载粒子动画。

**修改文件：** `frontend/src/layouts/FrontLayout.tsx`

**关键代码：**

```tsx
// 之前：静态导入，阻塞首屏
import ParticlesBackground from '@/components/ParticlesBackground';
import ParticleThemeSelector from '@/components/ParticleThemeSelector';

// 之后：动态导入，延迟加载
const LazyParticlesBackground = lazy(() => import('@/components/ParticlesBackground'));
const LazyParticleThemeSelector = lazy(() => import('@/components/ParticleThemeSelector'));

// 组件内：首屏渲染完成后再挂载粒子
const [showParticles, setShowParticles] = useState(false);

useEffect(() => {
  const id = requestAnimationFrame(() => {
    setTimeout(() => setShowParticles(true), 100);
  });
  return () => cancelAnimationFrame(id);
}, []);

// 渲染：条件渲染 + Suspense
{showParticles && (
  <Suspense fallback={null}>
    <LazyParticlesBackground isDark={isDarkTheme} />
    <LazyParticleThemeSelector isDark={isDarkTheme} />
  </Suspense>
)}
```

**效果：** 首屏 JS 从 1.86MB 降至 ~883KB，减少 **52%**。粒子动画在首屏内容渲染完成后才开始异步下载，用户无感知。

### 2. 首页 API 请求延迟（减少网络竞争）

**问题：** 首页在 `useEffect` 中立即发起 3 个并行 API 请求（文章列表、分类列表、标签列表），与 JS/CSS 资源争抢网络带宽，而首屏 Hero 区域完全不依赖这些数据。

**方案：** 使用 `requestIdleCallback` 将数据请求延迟到浏览器空闲时执行，优先保证首屏静态内容（Hero 区域）的快速渲染。

**修改文件：** `frontend/src/pages/home/index.tsx`

**关键代码：**

```tsx
// 之前：立即请求
useEffect(() => {
  fetchData();
}, []);

// 之后：等浏览器空闲再请求
useEffect(() => {
  if ('requestIdleCallback' in window) {
    const id = requestIdleCallback(() => fetchData());
    return () => cancelIdleCallback(id);
  } else {
    const timer = setTimeout(fetchData, 200);
    return () => clearTimeout(timer);
  }
}, []);
```

**效果：** Hero 区域立即渲染，API 请求不再与关键资源竞争带宽。

#### requestIdleCallback 详解

**是什么**

`requestIdleCallback` 是浏览器提供的一个 API，用来在「当前帧做完渲染、事件处理、布局/绘制等关键工作之后」，如果还有空闲时间，再执行你传入的回调。也就是说：**先保证页面流畅，再在“空档”里做不紧急的事**。

**执行时机**

- 浏览器每一帧（通常 16.6ms）里会做：处理输入、执行 JS、样式计算、布局、绘制、合成等。
- 一帧结束后，如果到下一帧开始前还有剩余时间（idle time），就会在此时执行 `requestIdleCallback` 的回调。
- 如果一直没有空闲（例如页面很忙或用户一直在操作），回调会被推迟，直到出现空闲或达到你设置的 `timeout`。

**API 形式**

```js
const id = requestIdleCallback(callback, options);
cancelIdleCallback(id);  // 取消
```

- **callback**：接收一个参数 `deadline` 对象：
  - `deadline.timeRemaining()`：当前帧剩余的大致毫秒数。
  - `deadline.didTimeout`：是否因为达到 `timeout` 而被强制触发。
- **options**（可选）：
  - `timeout`：毫秒数。若在 `timeout` 之前都没有空闲，也会在这一时间点强制执行一次回调（此时 `didTimeout === true`），避免任务被无限推迟。

**和 setTimeout 的区别**

| 对比项 | requestIdleCallback | setTimeout(fn, 0) |
|--------|---------------------|-------------------|
| 触发时机 | 本帧或后续帧的「空闲时段」 | 当前任务队列清空后的下一个宏任务 |
| 是否考虑渲染 | 会等布局、绘制等做完再跑 | 不关心，可能插在两次渲染之间 |
| 对首屏/交互的影响 | 尽量不抢关键工作的时间 | 可能和关键脚本、渲染抢时间 |
| 适用场景 | 分析、预加载、非关键请求等 | 需要“尽快执行但不阻塞”时 |

**在本项目中的用法**

首页需要请求文章/分类/标签数据，但这些只用于首屏下面的「精选文章」「最新发布」等区域，首屏 Hero 不依赖它们。因此：

1. 使用 `requestIdleCallback(fetchData)`：让浏览器先完成首屏的 JS 执行和渲染，再在空闲时发 API 请求，减少与关键资源的带宽和主线程竞争。
2. 兼容性：Safari 等部分浏览器不支持 `requestIdleCallback`，所以用 `'requestIdleCallback' in window` 判断，不支持时退化为 `setTimeout(fetchData, 200)`，用短延迟近似“等首屏忙完再请求”的效果。
3. 组件卸载时调用 `cancelIdleCallback(id)`（或 clearTimeout），避免组件已销毁仍执行回调。

**总结一句**：`requestIdleCallback` = “等浏览器这一阵子忙完了，有空再帮我跑这段代码”，适合首屏非关键、可延迟的数据请求，从而优化首屏性能。

### 3. 图片懒加载

**问题：** 首页文章卡片的封面图位于首屏以下（第三屏"最新发布"），但没有设置 `loading="lazy"`，浏览器会在首屏阶段就开始下载这些图片。

**方案：** 为所有文章封面 `<img>` 标签添加 `loading="lazy"` 属性。

**修改文件：** `frontend/src/pages/home/index.tsx`

**关键代码：**

```tsx
<img
  src={article.cover}
  alt={article.title}
  loading="lazy"
  className="w-full h-full object-cover ..."
/>
```

**效果：** 浏览器仅在图片即将进入视口时才发起请求，减少首屏不必要的网络开销。

### 4. 滚动事件 RAF 节流

**问题：** 首页使用全屏滚动（scroll snap），`scroll` 事件监听器每次触发都会调用 `setCurrentSection` 更新状态，导致频繁的 React 重渲染。

**方案：** 使用 `requestAnimationFrame` 节流滚动事件处理，确保每帧最多处理一次滚动计算。同时添加 `{ passive: true }` 标记提升滚动性能。

**修改文件：** `frontend/src/pages/home/index.tsx`

#### `{ passive: true }` 如何提升滚动性能

给 `addEventListener('scroll', handler, { passive: true })` 的核心作用是：**让浏览器确认你的监听器不会调用 `event.preventDefault()` 来阻止默认滚动行为**，从而可以更激进地优化滚动。

- **默认（非 passive）**：浏览器必须假设你的回调里可能会 `preventDefault()`，因此在滚动发生时，可能需要等待 JS 回调执行结果再决定是否滚动（尤其是在触摸/滚轮滚动相关事件中影响更明显），导致滚动与主线程 JS 竞争，出现卡顿风险。
- **passive: true**：你向浏览器承诺“不阻止默认滚动”。浏览器就可以 **立即开始滚动/合成**，同时异步执行回调，减少滚动被 JS 阻塞的机会，让滚动更顺滑。

**注意：** 设置为 `passive: true` 后，在回调里调用 `e.preventDefault()` 将不会生效（并可能在控制台看到警告）。本项目的滚动监听只用于计算当前 section，不需要阻止滚动，因此适合开启 `passive: true`。

**关键代码：**

```tsx
// 之前：无节流
const handleScroll = () => {
  const index = Math.round(container.scrollTop / height);
  setCurrentSection(Math.min(index, sections.length - 1));
};
container.addEventListener('scroll', handleScroll);

// 之后：RAF 节流 + passive
let rafId = 0;
const handleScroll = () => {
  if (rafId) return;
  rafId = requestAnimationFrame(() => {
    const index = Math.round(container.scrollTop / height);
    setCurrentSection(Math.min(index, sections.length - 1));
    rafId = 0;
  });
};
container.addEventListener('scroll', handleScroll, { passive: true });
```

**效果：** 滚动过程中 React 重渲染次数大幅减少，滚动更加流畅。

### 5. Hero 浮动粒子缓存

**问题：** Hero 区域的 20 个浮动粒子在每次组件渲染时都会重新调用 `Math.random()` 生成随机值，导致不必要的 DOM 重建。

**方案：** 使用 `useMemo` 缓存粒子数据，仅在组件首次挂载时生成。

**关键代码：**

```tsx
const heroParticles = useMemo(() =>
  Array.from({ length: 20 }, (_, i) => ({
    id: i,
    size: Math.random() * 6 + 2,
    left: `${Math.random() * 100}%`,
    top: `${Math.random() * 100}%`,
    delay: `${Math.random() * 2}s`,
  })), []);
```

**效果：** 避免每次渲染时创建 20 个新对象和重建对应的 DOM 节点。

## 三、部署配置优化

### 构建产物 Hash 化

在 `.umirc.ts` 中开启 `hash: true`，使构建产物文件名包含内容 hash（如 `umi.ab59b289.js`），解决部署后浏览器缓存导致的旧 chunk 加载失败问题。

### Nginx 缓存策略

更新 Nginx 配置，对 `index.html` 设置 `no-cache`，对带 hash 的静态资源设置 30 天长期缓存：

```nginx
# index.html 不缓存
location = /index.html {
    add_header Cache-Control "no-cache, no-store, must-revalidate";
}

# 带 hash 的静态资源长期缓存
location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf|eot)$ {
    expires 30d;
    add_header Cache-Control "public, immutable";
}
```
