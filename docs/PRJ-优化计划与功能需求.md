# 若风博客 - 优化计划与功能需求

> 基于现有功能分析，围绕 **用户体验**、**用户留存**、**性能优化**、**首屏优化** 四大维度制定
> 生成时间：2026-04-05

---

## 一、优化目标

| 维度 | 核心目标 | 衡量指标 |
|------|---------|---------|
| 用户体验 | 操作流畅、反馈即时、无障碍访问 | 交互响应 < 100ms、无死链/白屏 |
| 用户留存 | 让读者愿意常来、多看、多互动 | 回访率、人均阅读篇数、评论数 |
| 性能优化 | 资源体积小、请求少、渲染快 | Lighthouse Performance > 90 |
| 首屏优化 | 首屏内容秒级可见 | FCP < 1.5s、LCP < 2.5s |

## 二、用户体验优化

### UE-01：文章预加载 & 页面切换动画

**现状**：点击文章链接后白屏加载，等待数据返回才渲染内容。

**方案**：
- 鼠标悬停文章链接时，通过 `<link rel="prefetch">` 预加载目标页面的 JS chunk
- 点击后显示骨架屏过渡，数据到达后淡入真实内容
- 利用 Umi 的 `prefetch` 配置或手动 `import()` 预加载

**优先级**：高
**工作量**：2-3h

---

### UE-02：搜索体验增强

**现状**：搜索只在文章列表页可用，无实时反馈，需要按回车。

**方案**：
- 全局搜索快捷键 `Ctrl+K` / `Cmd+K` 唤起搜索浮层
- 输入即搜（debounce 300ms），显示文章标题 + 摘要预览
- 搜索历史记录（localStorage，最近 10 条）
- 搜索结果高亮关键词

**优先级**：高
**工作量**：4-5h

---

### UE-03：图片灯箱查看器

**现状**：文章中的图片无法放大查看，点击无反应。

**方案**：
- 点击文章内图片弹出灯箱，支持缩放、左右切换、键盘操作（Esc 关闭、方向键切换）
- 轻量实现，不引入第三方库，纯 CSS + JS 实现
- 支持手势缩放（移动端双指缩放）

**优先级**：中
**工作量**：2-3h

---

### UE-04：移动端手势支持

**现状**：移动端只能点击导航，无滑动手势。

**方案**：
- 文章详情页左右滑动切换上下篇
- 首页支持上下滑动切换 section（已有 scroll snap）
- 移动端底部固定导航栏（首页、文章、工具、我的）

**优先级**：中
**工作量**：3-4h

---

### UE-05：无障碍访问（A11y）基础

**现状**：缺少 ARIA 属性，键盘导航不完整。

**方案**：
- 为所有交互元素添加 `aria-label`
- 确保所有功能可通过 Tab 键访问
- 文章详情页添加跳转到主内容的快捷链接
- 颜色对比度检查，确保 WCAG AA 合规

**优先级**：低
**工作量**：3-4h

---

## 三、用户留存优化

### UR-01：个性化推荐 - 猜你喜欢

**现状**：文章详情页的"相关推荐"仅基于同分类/标签，无个性化。

**方案**：
- 基于用户阅读历史（localStorage 中已有），分析偏好分类和标签
- 首页新增"为你推荐"区域（根据偏好权重排序）
- 文章详情页优化相关推荐算法：同标签权重 > 同分类 > 热门
- 未登录用户使用热门口碑推荐

**优先级**：高
**工作量**：4-5h

---

### UR-02：阅读成就系统

**现状**：无激励用户持续阅读的机制。

**方案**：
- 阅读里程碑提示（首次阅读、阅读 5 篇、10 篇、连续 3 天访问等）
- 成就数据存 localStorage，达到时弹出轻量 toast 提示
- 个人中心展示阅读统计（总篇数、连续天数、偏好标签词云）
- 不需要后端支持，纯前端实现

**优先级**：中
**工作量**：3-4h

---

### UR-03：文章系列/专栏

**现状**：文章之间没有关联关系，系列教程难以串联阅读。

**方案**：
- 后端新增 `series` 字段（文章系列 ID + 排序序号）
- 文章详情页展示系列导航（上一篇/下一篇 + 系列目录）
- 后台管理支持创建系列并关联文章
- 首页/列表页展示系列入口

**优先级**：中
**工作量**：6-8h（含后端）

---

### UR-04：通知系统 - 评论回复提醒

**现状**：用户评论后无法知道是否被回复，没有互动闭环。

**方案**：
- 后端记录用户的评论和回复关系
- 前端导航栏新增通知图标（显示未读数）
- 支持标记已读、全部已读
- 通知类型：回复我的评论、文章被点赞（可选）
- 后端新增 `Notification` 模型和 API

**优先级**：高
**工作量**：6-8h（含后端）

---

### UR-05：RSS 订阅支持

**现状**：无 RSS 订阅，技术博客的核心受众习惯使用 RSS。

**方案**：
- 后端新增 `GET /api/feed/rss` 和 `GET /api/feed/atom`
- 输出最近 20 篇文章的 XML 格式 feed
- 前端页面展示 RSS 订阅图标
- 轻量实现，依赖零额外库

**优先级**：中
**工作量**：2h

---

### UR-06：文章目录 - 阅读进度同步

**现状**：TOC 目录不会高亮当前阅读位置。

**方案**：
- 使用 IntersectionObserver 监听标题元素进入视口
- TOC 中高亮当前阅读到的标题
- 点击 TOC 项平滑滚动到对应位置

**优先级**：高
**工作量**：2h

---

## 四、性能优化

### PF-01：关键路由预加载

**现状**：路由切换时才加载对应 JS chunk，有白屏间隙。

**方案**：
- 首屏加载完成后，在浏览器空闲时预加载高频路由 chunk（文章列表、文章详情）
- 使用 `<link rel="prefetch">` 或 `requestIdleCallback + import()`
- 配合 Service Worker 缓存已加载的 chunk

**优先级**：高
**工作量**：2h

---

### PF-02：接口数据缓存层

**现状**：每次访问页面都重新请求 API，无前端缓存。

**方案**：
- 前端实现轻量缓存层（基于 localStorage 或内存 Map）
- 缓存策略：文章列表缓存 5 分钟，分类/标签缓存 30 分钟
- 提供手动刷新机制（下拉刷新、缓存过期自动失效）
- 减少重复请求，秒开已访问过的页面

**优先级**：高
**工作量**：3-4h

---

### PF-03：Service Worker 离线缓存

**现状**：无离线访问能力，断网即白屏。

**方案**：
- 注册 Service Worker，缓存静态资源（JS/CSS/字体）
- 实现 App Shell 模式：HTML + 布局 JS 始终从缓存加载
- API 数据采用 Network First 策略（优先网络，失败用缓存）
- 离线时展示"当前处于离线模式"提示

**优先级**：中
**工作量**：4-5h

---

### PF-04：图片优化 - WebP + 响应式尺寸

**现状**：文章封面图使用原始格式，未做格式和尺寸优化。

**方案**：
- 后端上传时自动转换为 WebP 格式（保留原图作为 fallback）
- 使用 `<picture>` 标签：支持 WebP 的浏览器加载 WebP，否则加载原图
- 响应式图片：`srcset` 根据屏幕宽度加载不同尺寸
- 封面图生成缩略图（列表页用小图，详情页用大图）

**优先级**：中
**工作量**：4-5h（含后端）

---

### PF-05：组件渲染优化

**现状**：部分列表组件在数据未变化时仍会重渲染。

**方案**：
- 文章卡片组件使用 `React.memo` 避免不必要重渲染
- 列表页使用虚拟滚动（文章数量多时）
- 使用 `useCallback` / `useMemo` 优化事件处理函数
- React DevTools Profiler 分析热点组件

**优先级**：低
**工作量**：2-3h

---

### PF-06：Nginx 进一步优化

**现状**：已开启 gzip，但可进一步优化。

**方案**：
- 开启 Brotli 压缩（比 gzip 再小 15-20%）
- 开启 HTTP/2 Push（推送关键 CSS/JS）
- 静态资源 CDN 加速（可选：Cloudflare 免费 CDN）
- 开启 `etag` 精确缓存验证

**优先级**：低
**工作量**：1-2h

---

## 五、首屏优化

### FS-01：骨架屏精细化

**现状**：骨架屏为基础矩形占位，与实际内容布局差异较大。

**方案**：
- 首页骨架屏精确模拟 Hero + 文章卡片布局
- 文章列表骨架屏模拟筛选栏 + 卡片网格
- 文章详情骨架屏模拟标题 + 元信息 + 正文段落
- 骨架屏添加渐变闪烁动画，减少"跳变感"

**优先级**：高
**工作量**：3-4h

---

### FS-02：关键渲染路径优化

**现状**：CSS 通过 JS 动态加载，阻塞渲染。

**方案**：
- 提取首屏关键 CSS（Hero 区域、导航栏）内联到 HTML `<head>`
- 非关键 CSS 异步加载（`media="print" onload` 技巧）
- 字体使用 `font-display: swap` 避免文字闪现
- 确保 `umi.css` 不阻塞首屏渲染

**优先级**：高
**工作量**：2-3h

---

### FS-03：首屏数据 SSR/预渲染

**现状**：SPA 模式，首屏 HTML 为空壳，爬虫和用户都需要等 JS 加载。

**方案**（轻量方案，不改架构）：
- 使用 `prerender-spa-plugin` 或静态预渲染，为首页、文章列表等高频页面生成静态 HTML
- Nginx 判断 User-Agent，爬虫返回预渲染页面，普通用户仍走 SPA
- 或者：使用 Umi 的 `mpa` 模式为关键页面生成独立 HTML

**备选方案**（中量方案）：
- 首页和文章详情页使用 `renderToStaticMarkup` 做局部 SSR
- 后端 EJS 模板渲染首屏 HTML，JS 水合接管

**优先级**：中
**工作量**：6-8h

---

### FS-04：资源加载优先级控制

**现状**：所有资源同等优先级加载。

**方案**：
- 首屏关键 JS 添加 `<script>` 标签
- 非首屏资源使用 `<link rel="preload">` 控制加载时序
- 字体使用 `<link rel="preload" as="font" crossorigin>`
- 图片使用 `fetchpriority="high"` 标记首屏关键图片

**优先级**：中
**工作量**：1-2h

---

### FS-05：第三方脚本延迟执行

**现状**：百度推送脚本在 `<head>` 中同步执行。

**方案**：
- 百度推送脚本改为 `async` 或 `requestIdleCallback` 延迟执行
- 分析脚本（如有）延迟到用户交互后加载
- 粒子背景已是懒加载，继续保持
- 所有非首屏必需的脚本统一延迟策略

**优先级**：中
**工作量**：1h

## 八、验收标准

### 用户体验
- [ ] 所有页面切换有骨架屏过渡，无白屏闪烁
- [ ] 全局搜索 `Ctrl+K` 可用，300ms 内显示结果
- [ ] 文章内图片可点击放大
- [ ] TOC 目录实时高亮当前阅读位置

### 用户留存
- [ ] 首页展示"为你推荐"区域（基于阅读历史）
- [ ] 评论被回复时导航栏有通知提示
- [ ] RSS feed 可正常订阅
- [ ] 阅读成就 toast 正常触发

### 性能优化
- [ ] Lighthouse Performance 得分 > 90
- [ ] 重复访问页面使用缓存，< 500ms 渲染完成
- [ ] 离线时可访问已缓存的页面

### 首屏优化
- [ ] FCP < 1.5s（Fast 3G 网络）
- [ ] LCP < 2.5s（Fast 3G 网络）
- [ ] 首屏骨架屏与实际内容布局匹配度 > 80%
